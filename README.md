### 第二个部分：Agent/openclaw

### 第三个部分：图形界面
- 只负责交互和数据呈现。如果是要看日程类的，优先选列表式

### ddd(领域驱动设计)
- 领域 A 和 领域 B，可以看作两个独立领域，彼此不知对方存在。
- 那么跨领域的协调和查询，就由顶层的编排器AGENT（核心中枢） 负责。编排器负责同时下发任务给两个 Agent 。两个 Agnet 处理完任务后，写完 DB 后将结果返回回去。顶上的编排器 Agent 再呈现出一个结果给用户（用户请求在交互层）。
- 领域驱动的好处：
1. 就是可以独立部署，有的用户只做 A，有的用户只做 B，有的两个都需要的情况下我们各取所好 
2. A 和 B 实际上是不同的语义，管理 A 的时候是一套规则。而管理 B 的时候实际上是完全另一套规则。因此可以完全按照板块的业务需要去任意发挥而不用担心两者之间会相互污染。符合“高内聚，低耦合“，这样好处是往后可以用少量的成本就增加一个新的业务领域而不影响现有 agent 的表现，且与此同时表结构隔离
- 综上，ddd 是未来拓展升级的核心
###  Agent 架构
1. ReAct 架构反复推理循环推理循环，每次只做一件事，灵活但效率低。批量行为会使得时间成倍增加
2. 故计划于执行流水线：1.计划+执行步骤；2.模型统一推理。不论用户想要一个动作还是多个动作，都归结成一个计划。生成完计划后，大模型统一推理、统一解析时间、统一解析地点，推给用户后。用户只需要点击确认，大模型即可操纵数据库执行。
#### 消耗算力最多的其实是推理和执行计划的过程，一个计划哪怕包含许多步骤，在数据库层面都是高效的
### smart agent, dumb tool
- 即充分相信大模型的推理和执行能力，因此要把工具做到极致和简单，高强度复用。把增删改查这些功能简化为 executor，即执行器。而校验功能等复杂功能都交给 agent 和大模型，它们掌握充分的上下文信息，决定什么时候 reason 什么时候 act，得到计划后用“dumb tool“去做最 dumb、最 simple的执行

## 工程实践
### 可观测性和功能品质问题
代码里应有 20% 是与功能无关的，主要聚焦于可观测性和功能品质问题。

可观测性：决策id追踪系统，即用 logging 埋点上报。大模型在搜的过程中，直接锁定关键 log、直接锁定关键结果，这样可以帮助大模型迅速解决问题

工程品质问题：自动优化飞轮（eval+traceid+flywheel），即
打造“生产数据“、“用例生成“、“决策 id 追踪“、“根因锁定“这四者的闭环循环（只要有 bug，查 log；把 bug 写成测试用例，在本地复现 bug），做成自动化流程，通过 case 来做回归测试

## 技术公开(multi-agent-system)
### 是否过度设计？
- 项目愿景：取决于项目愿景。是想做个人小工具，还是达到真正投放市场。以及产品未来的上限设计在哪？
- 前瞻性：如果小工具，能 work 就行。一旦涉及到市场、生产，就必须把架构设计得 有前瞻性。这是一个在设计端、产品端要想好的问题
### 设计体系
- 在整个系统中采用 tokenized 的设计体系。字体、字号、颜色、spacing，全部变成 design token 这套设计体系，通过 .json 加载到每个组件中。好处是一旦想做界面、想做组件，直接读取按照这套规范就能生成，而不用重新设计。（可采用 pencilmcp+codex/claude+mobbi）
- 不建议“原创“，因为在当今互联网时代，交互基本已定型了，就那么几种方式（因为其它的都被淘汰了）。无非品的就是品味(taste)，哪个颜色好看、哪个布局更合理、更符合审美和口味，要向行业成功案例看齐
- 技术栈选型的原则是“该省省，该花花“，在关于产品未来和前途命运的关键领域一定要选择一步到位。agent 的框架和 sdk 可以选 vercel sdk,可解决大量的摸索和探索时间。第二重要的是数据库这块，大模型、agent、数据这三方面倘若不做足够的预留，那会留下一大笔技术债。选 docker 还是 k8s，选 native 还是 web，都有考究
- api 选择举例：sonnet、高德 api、向量库用千问embedding、jina reranker
