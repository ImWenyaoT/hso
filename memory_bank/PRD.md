# PRD: The Haiku, The Sonnet and The Opus (HSO)

## 1. 产品一句话

HSO 是一个帮助科研新手以更低门槛完成论文项目的工作台。

它不是泛化的“科研万能助手”，也不是第一天就追求完整替代 Overleaf 或 TeXstudio 的全功能编辑器。HSO 的第一版主线很明确：让用户从一个模糊课题或一篇种子论文出发，逐步进入一个可继续推进的论文项目，并完成模板化排版、构建与预览。

## 2. 产品定位

HSO 的核心定位是一个以 LUI（语言交互界面）为统一入口、以论文项目为核心对象的低门槛桌面论文工作台。

这个定位包含三个关键判断：

1. 对外主叙事是“低门槛完成论文项目”，而不是“双主线科研平台”
2. LaTeX 构建域是第一版主价值域，信息检索域是前置辅助域
3. 产品强调结构化工作流，而不是强调用户必须掌握 LaTeX 语法细节

## 3. 目标用户与核心问题

HSO 面向的不是已经拥有稳定科研方法论的资深研究者，而是刚进入科研训练阶段的研究生 noob 和 beginner。

这类用户常见的困境包括：

1. 导师给了论文或方向，但用户不知道该从哪里开始
2. 用户能看懂一些材料，但无法快速形成结构化的研究准备结果
3. 用户进入论文写作阶段后，会被模板、图表、章节组织和编译报错卡住
4. 用户真正想做的是表达内容与结构，而不是学习大量 LaTeX 工程细节

第一版产品要解决的核心问题是：

1. 帮用户把“模糊的起点”转成“可继续推进的论文项目输入”
2. 帮用户以更低门槛进入论文写作与排版工作流
3. 帮用户在受控范围内完成模板套用、结构化编排、构建预览与常见报错处理

## 4. 第一版产品主线

HSO 第一版围绕一条简单主线展开：

`从模糊研究起点进入系统 -> 沉淀研究材料 -> 建立论文项目 -> 完成结构化排版与构建预览`

这条主线意味着：

1. 论文项目是第一版最重要的产品对象
2. 信息检索域的意义在于把用户送进论文项目，而不是独立承担产品主叙事
3. 产品价值最终要落在“论文项目能否被顺利推进”上，而不是单独的检索结果是否丰富

## 5. 核心对象：论文项目

第一版中，HSO 最重要的对象不是研究卡片，也不是资料收藏夹，而是论文项目。

论文项目用于组织用户围绕一篇目标论文持续推进的核心资产，至少包括：

1. 目标模板或出版社格式
2. 章节结构
3. 图表与素材
4. 参考资料与引用来源
5. 构建结果与预览状态

在产品关系上：

1. 研究卡片是论文项目的前置输入材料
2. 资料库和收藏夹是论文项目的辅助资源层
3. 论文项目才是第一版中真正承载主工作流的核心对象

## 6. 领域划分

HSO 内部仍然保持两个独立领域，但两者在产品叙事中的权重不同。

### 6.1 信息检索域

信息检索域负责“写作前”的研究准备阶段，其目标不是替用户做主观学术判断，而是帮助用户看到从哪里开始。

第一版中，它的主要职责是：

1. 接收关键词或种子论文作为输入
2. 提供客观、量化、可验证的研究材料
3. 将结果沉淀为研究卡片或候选资料
4. 支持一键进入论文项目

它的边界是：

1. 提供材料，不代替用户做选题判断
2. 提供客观信息，不输出“这个方向一定值得做”之类的主观结论
3. 不承担论文正文写作主流程

### 6.2 LaTeX 构建域

LaTeX 构建域是第一版的主价值域，也是 HSO 对外最核心的产品能力。

它负责帮助用户在不深度学习 LaTeX 细节的前提下，完成论文项目中的结构化排版工作。

第一版中，它的主要职责是：

1. 创建和管理论文项目
2. 套用模板或出版社格式
3. 组织章节、图表和素材
4. 进行构建与预览
5. 在受控范围内解释和修复常见编译问题

## 7. 第一版核心能力

围绕“论文项目”这一核心对象，第一版聚焦以下能力：

1. 从关键词或种子论文进入系统
2. 生成研究卡片，并支持转入论文项目
3. 为论文项目套用模板
4. 管理章节结构、图表素材和参考资料
5. 支持一键构建与预览
6. 对系统生成内容中的常见编译错误提供解释、定位、修复建议和有限自动修复

这些能力共同构成一个“轻闭环”的论文工作台，而不是一个泛化编辑器。

## 8. 第一版范围与非目标

为了避免过度设计，第一版明确不追求以下目标：

1. 不替代全功能 LaTeX IDE 或自由编辑器
2. 不承诺兼容任意复杂、任意风格的自定义 LaTeX 工程
3. 不替用户做主观学术判断或选题决策
4. 不把润色、扩写、改写作为第一版主叙事
5. 不把信息检索域做成独立于论文项目之外的完整产品线

润色与扩写能力可以存在，但只应被定义为辅助能力，而不是第一版核心工作流。

## 9. 交互方式

HSO 采用“LUI 前置，GUI 辅助”的交互方式，并以桌面应用作为第一版默认载体。

这意味着：

1. 用户可以直接通过自然语言描述任务进入系统
2. 图形界面负责快捷入口、结构化呈现和清晰反馈
3. 列表型、资料型、项目型信息优先使用直观的结构化展示
4. GUI 不承担复杂业务决策，主要承担交互与呈现
5. 第一版默认通过 `.app` / `.exe` 形式启动，而不是要求用户访问网页站点
6. 核心项目管理、结构编辑、历史结果查看应尽量支持本地优先使用
7. 视觉与组件体系应从一开始采用 design token 思路，而不是后期补做

## 10. 系统原则

第一版 PRD 只保留与产品表达直接相关的系统原则，不展开过多实现细节。

### 10.1 Thin Client

这里的 `thin client` 指的是“桌面端不承载复杂业务决策”，而不是要求产品必须是浏览器网页。

第一版中，桌面客户端仍然主要负责：

1. 交互
2. 数据呈现
3. 本地文件与项目入口

核心逻辑仍然放在 Agent / Application 层。

### 10.2 DDD

信息检索域与 LaTeX 构建域是两个彼此独立的领域。

这样做的原因是：

1. 两个领域语义不同，规则不同
2. 各自可以独立演进，互不污染
3. 用户可以只使用其中一个领域
4. 后续新增领域时成本更低

### 10.3 顶层编排

跨领域协同由顶层编排器负责，而不是让两个领域直接相互耦合。

典型场景包括：

1. 把研究卡片转成论文项目
2. 在论文项目中回查资料或补充素材

### 10.4 Smart Agent, Dumb Tool

系统尽量把复杂逻辑放在 Agent 和模型侧，把工具做成简单、稳定、可复用的执行器。

也就是说：

1. Agent 负责理解上下文、归纳计划、决定何时 reason 与 act
2. Tool 负责执行简单明确的动作，例如增删改查和构建执行

### 10.5 Design Token System

第一版虽然以产品主流程为先，但界面体系不应完全临时拼装。

这意味着：

1. 颜色、字号、间距、圆角、层级等应尽量抽象成 design token
2. 组件实现应优先消费 token，而不是把样式值散落在页面里
3. 第一版不要求完整设计系统文档，但要求为后续演进保留一致的设计语言基础

## 11. 产品流程示意

从用户视角看，第一版的典型流程如下：

1. 用户输入一个方向关键词，或输入一篇种子论文
2. 系统返回研究材料与研究卡片
3. 用户确认后，将结果转入一个论文项目
4. 用户在论文项目中选择模板、组织章节、管理图表和资料
5. 系统完成构建与预览，并在受控范围内处理常见报错

在运行形态上，这个流程默认发生在本地启动的桌面应用中。联网能力用于增强研究检索、模型能力与后续扩展，而不是作为第一版使用门槛。

## 12. 后续演进方向

第一版之后，HSO 可以继续扩展，但这些内容不应反向绑架第一版设计。

可预留的演进方向包括：

1. 更丰富的研究准备能力
2. 更强的模板与期刊适配能力
3. 更完善的项目协作能力
4. 更成熟的工程可观测性与自动优化飞轮
5. 在桌面版稳定后，再评估是否需要单独的 Web 配套入口
6. 在产品与数据边界稳定后，将单编排器逐步演进为更明确的 multi-agent 体系

这些方向说明产品有扩展空间，但不作为当前 PRD 的核心承诺。
