# Lessons

## 2026-03-28

- 当用户指出 `memory_bank` 文档需要回到 `idea.md` 的真实愿景时，必须逐项核对是否发生了"愿景降级"。
- 如果 `idea.md` 提到更强的架构方向，例如 multi-agent、design token、自动优化飞轮，不能只在 `memory_bank` 里保留弱化后的抽象表述。
- 做文档收敛时，要明确区分三类状态：`v1 必做`、`中长期方向`、`暂不纳入`，避免把中长期方向错误删除。
- 当用户逐条回答实施计划中的澄清问题时，应立即把这些回答视为新的冻结约束，并基于这些约束收敛推荐方案，而不是继续重复提出同一层级的开放问题。
- 当用户给出最终实现口径时，不要停留在"这已经够清楚了"的评价层，应尽快把这些口径写回文档，避免同一问题在后续再次出现。
- 当技术栈已经通过多轮讨论基本确定时，不能把"主编程语言"留在默认推断层，必须显式写进文档，尤其要区分主语言与辅助脚本语言。

## 2026-03-29

- **规格够了就写代码，不要再写规划文档**。当产品契约、领域模型、UI 规划、输入流程、转化流程都已经冻结且有文档记录，下一步是写代码（建表、IPC、页面骨架、Agent），而不是再出一份"实施计划规划文档"。
- **`implementation_plan.md` 的陷阱**：将任务设计成"规划 X 流程"（定义、记录、审查）= 产出更多文档，永远不会产出代码。任务必须是"实现 X 文件"、"写 Y 函数"、"在 Z 位置建表"。
- **用 Write 工具前必须先用 Read 工具**：直接 Write 一个从未 Read 过的文件会报错"File has not been read yet"，必须先 Read 再 Write。
- **文档描述的项目状态必须和实际状态一致**：CLAUDE.md 说"规划/规格阶段，尚无实现代码"、AGENTS.md 说"当前尚无已搭建的运行时"，但实际上代码已经写了。文档过时会误导后续的 AI 工作。每完成一个编码阶段，及时更新 CLAUDE.md、AGENTS.md、implementation_plan.md 中的状态描述。
